## 3. 我们是如何完成这个项目的（过程视角）

### 3.1 先文档驱动，再编码
先基于 PRD + TECH_SPEC + API 合同文件统一业务口径，先把“做什么、怎么验收、边界怎么处理”对齐，再进入编码。

### 3.2 采用分层渐进实现
按“后端先可用 -> 前端联调 -> 契约校验 -> 工程化补齐（CLI/skills）-> 启停与运行验证”的顺序推进，减少互相阻塞。

### 3.3 全程以可测试为约束
每个阶段都补对应自动化验证（单测/集成/接口脚本），出现异常时优先复现、再修复、最后回归。

### 3.4 最后回到运行态质量
完成编码后，重点处理了运行中暴露的真实问题：代理配置、接口完整性、默认数据、字符编码、服务启停。

## 4. 编码过程中遇到的问题与解决方式

### 问题 1：前端请求 `/api` 返回 `text/html`
- 现象：接口响应不是 JSON，而是前端页面 HTML。
- 根因：Vite dev server 未配置 `/api` 代理，命中 SPA 回退。
- 处理：在 `apps/web/vite.config.ts` 增加 `server.proxy`，将 `/api` 转发到 `http://127.0.0.1:3000`。
- 结果：`Content-Type` 恢复为 `application/json`。

### 问题 2：新增能力接口“有路由但不完整”
- 现象：路由存在，但关键业务校验/错误码分支不全。
- 根因：早期实现偏“可跑通”，未完全落实 TECH_SPEC 的新增能力约束。
- 处理：补齐参数校验、类型限制、分类存在校验、URL 安全、重名校验；并新增自动化测试。
- 结果：新增能力接口按规范返回 `CAP_4001/4002/4003/4006`，成功返回 `0`。

### 问题 3：默认伪数据引入后影响测试
- 现象：部分用例出现重复名冲突，导致预期失败。
- 根因：`createApp({ categories: [...] })` 这类测试场景仍被自动注入了默认 capabilities。
- 处理：改为仅 `createApp()` 无 options 时启用 seed；传了任意 options 则不自动补默认数据。
- 结果：测试恢复稳定，开发环境仍能开箱看到伪数据。

### 问题 4：脚本替换导致 `app.js` 短暂语法损坏
- 现象：批量替换时局部函数头丢失。
- 根因：字符串替换方式脆弱。
- 处理：立即整文件重写为正确版本，并重新跑全量测试。
- 结果：服务恢复，回归通过。

### 问题 5：服务端中文文本乱码（message/数据字段）
- 现象：`message` 显示乱码。
- 根因：源文件编码不统一（非 UTF-8 内容被错误解码）。
- 处理：统一 `apps/server/src/**/*.js` 为 UTF-8，重启服务并做码位级验证。
- 结果：中文返回恢复正常。

### 问题 6：本地命令在沙箱下出现 `EPERM` /权限受限
- 现象：某些 dev 启动或 one-off 脚本在沙箱环境无法正常 spawn。
- 根因：沙箱权限限制。
- 处理：使用提权执行（require_escalated）完成关键命令。
- 结果：启动、验证、重启均可执行。

## 5. 复盘：如何和我沟通，能减少上述问题

### 5.1 在实现前固定“强约束清单”
建议在任务开始就明确：
1. 必须严格遵守的接口契约（字段、错误码、HTTP 状态）
2. 必须覆盖的测试清单
3. 开发态是否需要默认 seed 数据，以及 seed 生效条件

### 5.2 明确“先给修改清单，再执行”
对高风险修改（路由主干、配置、编码、批量替换）可要求：
- 先输出“将改哪些文件、为什么改、预期影响”
- 你确认后再动手
这样可以显著降低第 3/4 类问题。

### 5.3 对环境类要求一次性说全
例如：
- dev 端口约定（前后端）
- 是否必须走代理
- 文件编码统一为 UTF-8
- 是否允许提权运行命令
提前讲清可避免第 1/5/6 类问题。

### 5.4 要求“每步交付都带验证证据”
可固定让每次交付都附：
1. 变更文件列表
2. 执行的测试命令
3. 关键断言结果
可更早发现“接口有路由但语义不完整”的偏差。

### 5.5 对“仅开发便捷”的改动设开关
像默认伪数据，最好一开始就指定为：
- 仅 dev 生效
- 测试和自定义初始化禁用
可避免对现有用例造成隐性影响。
