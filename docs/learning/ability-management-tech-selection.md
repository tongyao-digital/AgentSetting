# 能力管理需求技术选型说明（React + TS / Node.js）

## 1. 结论
本需求属于常规后台管理系统，采用前端 `React + TypeScript`、后端 `Node.js` 是合理且可落地的技术选型。

## 2. 选型判断
1. 前端为表单、列表、筛选、权限显隐、上传等典型后台交互，`React + TypeScript` 适配度高。
2. 后端核心是接口编排、外部 HTTP 调用、同步任务与权限控制，属于 I/O 密集场景，`Node.js` 适配度高。
3. 团队若已有 JavaScript/TypeScript 经验，该组合能降低沟通和维护成本。

## 3. 前端建议（React + TypeScript）
1. 强类型建模：对能力、分类、请求配置、错误码建立统一类型。
2. 表单与校验：统一输入校验规则（长度、唯一性、URL 合法性、超时范围）。
3. 状态管理：按模块拆分（分类管理、能力管理、同步状态）。
4. 权限控制：前端只做显示控制，后端必须做强校验。
5. 测试：覆盖关键流程的 UI 自动化与接口联调。

## 4. 后端建议（Node.js）
1. 建议直接使用 TypeScript，减少前后端字段漂移。
2. API 层与同步任务分离：接口服务与定时同步作业分职责部署。
3. 幂等与并发：编辑使用乐观锁（409），删除接口保持幂等。
4. 外部调用安全：限制 URL 范围，拦截内网地址，敏感信息加密存储与脱敏日志。
5. 可观测性：为同步任务和外部调用提供 trace_id、耗时、失败原因。

## 5. 主要风险与控制点
1. 风险：同步覆盖人工修改导致数据冲突。
   控制：明确“同步字段优先级”和冲突日志。
2. 风险：重复提交、并发编辑导致脏写。
   控制：乐观锁 + 幂等键。
3. 风险：外部 HTTP 不稳定导致能力不可用。
   控制：超时、重试、熔断、告警。
4. 风险：仅前端控权造成越权操作。
   控制：后端 RBAC 强校验。

## 6. 是否需要更换栈
当前阶段不建议更换技术栈。优先把工程约束补齐：
1. 类型统一（前后端 TypeScript）。
2. 同步与审计机制。
3. 自动化测试和发布门禁。

## 7. 最小落地建议
1. 前端：React + TypeScript + 表单校验 + UI 自动化。
2. 后端：Node.js + TypeScript + RBAC + 同步任务 + 审计日志。
3. 质量门禁：P0 用例 100% 通过后发布。
